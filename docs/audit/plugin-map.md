## Webhook receiver — `class-webhook-handler.php`

Webhook is the most exposed PUBLIC entry point. Three verification paths visible:

| Mechanism | Line | Use case |
|---|---|---|
| `hash_equals( $expected_token, $token )` | 229 | Token-based auth (Plex Pass, configured per-service) |
| `hash_equals( 'sha256=' . hash_hmac(...), $signature )` | 275 | HMAC-SHA256 (e.g. Trakt-style signed bodies) |
| `hash_equals( $expected_user, $username ) && hash_equals( $expected_pass, $password )` | 310 | HTTP basic-auth (Jellyfin) |

✅ All three use `hash_equals()` — timing-safe comparison.
✅ `hash_hmac('sha256', $body, $secret)` — proper HMAC.

Phase E Session E3 will: (a) test signature mismatch / replay / oversized body / malformed JSON for each mechanism, (b) verify replay protection (Section 7.2.3 critical finding trigger if missing).

`api_credentials` option do NOT explicitly set `autoload=false`. Default WP
behavior autoloads on every page request, exposing the option set to anyone
with shell access to dump `wp_options`. Tokens should be set with explicit
`'no'` autoload.

## SQL — `$wpdb` direct queries

10 direct `$wpdb` calls across 2 files:

**`includes/class-import-manager.php`** (lines 1449-1495):
✅ Uses `$wpdb->prepare()` correctly with `LIKE %s` placeholders.
✅ Uses `$wpdb->esc_like()` to escape the prefix string.

**`includes/admin/class-admin.php`** (lines 999-1016):
⚠️ Four `$wpdb->query()` calls with HARDCODED SQL strings (transient cache cleanup).
- No user input — the `LIKE` patterns are constants (`'_transient_postkind_indieweb_api_%'`).
- NOT a SQL injection vuln, but violates WP coding standard "always use prepare for queries that build SQL."
- Phase D Session D7 (dead code / WPCS triage) — flag for cleanup with `prepare()` even though no input flows through.

## `unserialize` / `maybe_unserialize`

5 occurrences:

**Risk-bearing:**
- `includes/class-import-manager.php:1461, 1491` — `maybe_unserialize($row->option_value)` on results from `$wpdb->get_results()`. The data is option_value from the `wp_options` table where job state is stored. **Risk:** if an attacker can write to `wp_options` (admin user, SQL injection elsewhere, or compromised database), they can plant a malicious serialized payload that triggers RCE on the next import-status query. Phase E Session E7 — investigate; mitigation could be `json_decode` round-trip on write.

**Safe:**
- `includes/class-plugin.php:248`, `includes/class-abilities-manager.php:95`, `includes/abilities/class-lookup-abilities.php:81` — `throw new Exception('Cannot unserialize singleton')`. Standard singleton-protection pattern.

## Three biggest unknowns before touching a file

1. **Why are the 7 extra kinds present** beyond the audit prompt's 17 (`note`, `article`, `event`, `photo`, `video`, `review`, `recipe`)? Are they intentional (PKIW ships richer than the legacy plugin), or migration leftovers from `dshanske/indieweb-post-kinds`? Determines D2 (legacy parity) scope.

2. **The 38-route REST surface is much larger than typical plugins.** Why? Hypothesis: each integration service has its own routes (settings/test/connect/disconnect × 12 services = ~48 routes), plus webhook endpoints, plus AI routes. If correct, the per-route `permission_callback` audit at E4 is the longest single security-audit session.

3. **The 4 `$wpdb->query()` calls without `prepare()` in class-admin.php** — even though they look safe (no user input), why are they present at all? Either (a) historical "we wrote this when no one cared," (b) deliberate (transient cleanup is hot-path and prepare overhead bothered the author), or (c) someone copy-pasted from a tutorial. Determines D7 (WPCS) cleanup approach.
