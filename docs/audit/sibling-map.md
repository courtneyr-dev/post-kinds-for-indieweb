### Composition concern — `class-pfbt-format-mf2.php` emits h-entry

**Per audit prompt Section 8.2:** PFBT must never emit h-entry. Post Kinds
owns h-entry when a kind is set; theme owns it otherwise.

**Current PFBT behavior:**

`includes/mf2/class-pfbt-format-mf2.php:78-140` defines a per-format mf2 map
where every entry includes `'entry_class' => 'h-entry'`:

```php
'aside'    => [ 'entry_class' => 'h-entry', ... ],
'gallery'  => [ 'entry_class' => 'h-entry', ... ],
'link'     => [ 'entry_class' => 'h-entry h-cite', ... ],
'quote'    => [ 'entry_class' => 'h-entry h-cite', ... ],
'reply'    => [ 'entry_class' => 'h-entry h-cite', ... ],
... etc.
```

Lines 169-175 register hooks:

```php
add_filter( 'post_class',  [ $this, 'add_mf2_post_classes' ], 10, 3 );
add_filter( 'the_content', [ $this, 'wrap_content_mf2' ], 5 );
add_filter( 'the_author',  [ $this, 'wrap_author_hcard' ] );
```

**Mitigation already in place:** the entire `init_hooks()` is gated behind
`PFBT_Feature_Flags::has_indieweb()` (line 165). So h-entry only emits when
IndieWeb integration flag is on. On a fresh PFBT install with no PKIW
present, no h-entry double-emission.

**Phase F1 fix required when PKIW is also active:**

```php
// In add_mf2_post_classes() / wrap_content_mf2(), before adding 'h-entry':
if ( apply_filters( 'pkiw_owns_h_entry', false, $post_id ) ) {
    // Skip h-entry — PKIW owns it for this post.
    $entry_class = str_replace( 'h-entry', '', $config['entry_class'] );
}
```

PKIW would register the filter as:

```php
add_filter( 'pkiw_owns_h_entry', function( $owns, $post_id ) {
    return ! empty( wp_get_post_terms( $post_id, 'kind', [ 'fields' => 'slugs' ] ) );
}, 10, 2 );
```

PKIW currently does NOT register `pkiw_owns_h_entry` (verified — no grep
match in PKIW source). So h-entry is currently emitted by **two** sources
when both plugins are active and PFBT's IndieWeb flag is on. Reproduce in
Phase F1 via mf2 validator on a seeded post.

### XFN role in composition

**XFN modifies `<a href="...">` tags inside rendered block content** to add
`rel="..."` attributes based on per-link metadata stored on the post.

**Composition implications:**

1. **No h-entry conflict** — XFN doesn't touch `post_class`. Compatible with
   both PKIW-owned and PFBT-owned h-entry strategies.

2. **No mf2 conflict** — XFN adds `rel` only. mf2 properties (`u-bookmark-of`,
   `u-in-reply-to` etc.) live on `class` attributes, not `rel`. Both can
   coexist on the same `<a>`.

3. **Risk: PFBT's `wrap_content_mf2` filter at `the_content` priority 5
   runs BEFORE XFN's `render_block` filter at priority 10.** PFBT may
   rewrite anchor markup in ways that strip rel attributes if its mf2 wrapper
   regenerates the HTML. **Phase F audit case:** seed post 37415
   (`overlap-bookmark-link-xfn`) — Bookmark + Link format + XFN-tagged
   anchors. Verify XFN's rel survives PFBT's content wrapping.

4. **Filter rel atomicity:** The XFN `process_block` filter operates on
   rendered block content, so it sees the post AFTER PFBT's `the_content`
   chain runs. XFN appends to existing `rel` values (per content-scanner
   logic) rather than replacing — so non-destructive on rel attrs already
   present from theme markup. **Phase F1 finding:** confirm XFN preserves
   rel values added by PFBT or theme, doesn't strip them.

## Three-plugin filter / action priority map

| Hook | Priority | Plugin | Purpose |
|---|---|---|---|
| `the_content` | **5** | PFBT (mf2 wrapper) | Wraps content with `<div class="e-content">` etc. — runs FIRST |
| `the_content` | 10 (default) | WP core formatting | wpautop, shortcodes, etc. |
| `render_block` | 10 | PFBT (post-kinds-integration & format-classes) | Various |
| `render_block` | 10 | XFN (`process_block` + `xfn_render_embed_block`) | Adds rel attributes |
| `post_class` | 10 | PFBT (`add_mf2_post_classes`) | Adds `h-entry` etc. — composition risk |
| `post_class` | 10 | PFBT (`filter_post_class` from format-classes) | Adds `pfbt-format-{slug}` etc. |
| `body_class` | (default) | PFBT (`filter_body_class`) | Adds `pfbt-format-{slug}` on singular |
| `set_post_format` | 10 | PFBT (`suggest_kind_on_format_change`) | Cross-suggests kind |
| `set_object_terms` | 10 | PFBT (`suggest_format_on_kind_change`) | Cross-suggests format |
| `init` | 5 | PKIW (`Taxonomy::register_taxonomy`) | Kind taxonomy register |
| `init` | 5 | PKIW (`Post_Type::maybe_register_post_type`) | Conditional CPT register |
| `init` | 20 | PFBT (`register_kind_meta`) | Post-meta for kind data PFBT consumes |
| `wp_enqueue_scripts` | (default) | XFN | Frontend script for interactivity |
| `enqueue_block_editor_assets` | (default) | All three | Editor JS bundles |

**Order risks for Phase F1:**

- PFBT's `wrap_content_mf2` at `the_content` priority 5 runs before WP
  core formatting. If it adds h-entry via DOM rewrite, that h-entry sticks
  even if `post_class` decides NOT to add it. Two sources of truth.
- XFN's render_block at priority 10 sees content AFTER PFBT's wrapper. So
  if PFBT strips a rel during wrap, XFN can't restore it from upstream.
- The `pkiw_owns_h_entry` filter must be registered AT or BEFORE both PFBT
  hooks fire. PKIW activation order (XFN → PFBT → PKIW) means PKIW
  registers last — but `init`-time registration runs before any
  `the_content` filter fires, so timing is safe.

### Naming inconsistency surfaced

Three different prefix conventions in use across the three plugins:

- **PFBT:** `pfbt_*` consistently (clean, short).
- **XFN:** `xfn_*` for the one filter it exposes (clean).
- **PKIW:** mixed — `pkiw_*` (2 filters, 1 action) AND `post_kinds_indieweb_*` (13 filters, 6 actions). The long prefix is the historical style; the short `pkiw_*` style is newer.

**Phase F3 recommendation:** standardize PKIW on `pkiw_*` for new
composition hooks (`pkiw_owns_h_entry`, `pkiw_target_url_resolved`,
`pkiw_kind_set`). Keep existing `post_kinds_indieweb_*` long-prefix hooks
for backward compatibility — too many themes/plugins likely listen to them.
Document both in `HOOKS-REFERENCE.md` with deprecation notes if needed.
