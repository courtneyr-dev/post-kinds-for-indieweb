# Post Kinds for IndieWeb — Design Tokens Reference

Generated by audit Session C2, 2026-05-01.

This is the authoritative per-token reference. Every token defined in
`styles/kind-tokens.css` is documented here with three columns: **token
name**, **what it paints** (which CSS property in which selector), and
**what happens if you don't set it** (the neutral default and what the
result looks like to the user).

The hard contract from Section 2 of the audit prompt:

> Plugin owns layout. Theme owns paint. The plugin's structural CSS may
> only reference colors and fonts through `var(--pkiw-*, neutral-fallback)`.
> If a theme sets the variable, the plugin uses it. If not, the kind
> inherits the theme's normal styling. **No kind ever forces a color the
> theme didn't ask for.**

If you remove all `--pkiw-*` variable definitions from your theme, the
plugin still renders correctly — every card surface inherits theme
typography, has no painted background, and uses the theme's accent color
where contrast is needed. That's the intended baseline.

---

## How to use this catalog

### From a theme

Set tokens in your theme's `:root` or in section-scoped selectors:

```css
:root {
  --pkiw-listen-card-bg: var(--wp--preset--color--surface);
  --pkiw-listen-card-accent: var(--wp--preset--color--accent);
  --pkiw-star-rating-fill: gold;
}

/* Per-context override */
.is-style-condensed .pkiw-listen-card {
  --pkiw-listen-card-bg: transparent;
}
```

### From a plugin or pattern

Always reference tokens via `var()` with the documented neutral fallback:

```css
.pkiw-listen-card {
  background: var(--pkiw-listen-card-bg, transparent);
  color:      var(--pkiw-listen-card-fg, inherit);
}
```

If the fallback in the consumer's CSS doesn't match the default in
`kind-tokens.css`, the documented default wins (because tokens cascade
from the layer registration).

## Structural tokens (carry real values)

Layout breaks without sizing values. These are NOT neutral — they carry
opinionated sizing that the plugin's grid logic depends on. Themes can
still override them when needed.

| Token | Default | Paints |
|---|---|---|
| `--pkiw-card-gap` | `1em` | Gap between siblings inside a card. Generic gutters. |
| `--pkiw-card-padding` | `0` | Inside-of-card padding. `0` so themes have full control. |
| `--pkiw-artwork-ratio-square` | `1 / 1` | Square aspect ratio fallback. Used by listen, jam, favorite, wish, mood. |
| `--pkiw-artwork-ratio-poster` | `2 / 3` | Movie poster ratio. Used by watch. |
| `--pkiw-artwork-ratio-wide` | `16 / 9` | Wide ratio. Available for video card kinds. |
| `--pkiw-artwork-ratio-book` | `2 / 3` | Book cover ratio. Used by read. |
| `--pkiw-star-size` | `1em` | Star glyph size. Defaults to current font size. |
| `--pkiw-star-count` | `5` | Number of stars in the rating. Themes can change to `10` for /10 ratings. |

## Shared card paint tokens (default neutral)

These are the base tokens. Per-kind tokens default to these, so themes
can paint cards uniformly by setting only `--pkiw-card-*`, OR per-kind
by also setting `--pkiw-{kind}-card-*`.

| Token | Default | What you see if you don't set it |
|---|---|---|
| `--pkiw-card-bg` | `transparent` | No painted card background. Card text appears directly on theme body background. |
| `--pkiw-card-fg` | `inherit` | Card text uses surrounding text color (theme body text). |
| `--pkiw-card-accent` | `currentcolor` | Decorative accents (eyebrow, separator) match surrounding text. |
| `--pkiw-card-border` | `transparent` | No visible card border. |
| `--pkiw-card-border-radius` | `0` | Sharp corners. |
| `--pkiw-card-shadow` | `none` | No drop shadow. |
| `--pkiw-card-text-secondary` | `inherit` | Captions, dates, byline text — same color as body text by default. |

## Star rating tokens (Section 5.4 special case)

The rating component currently hard-codes `#ffc107` (gold) for filled
stars. Token contract: filled stars inherit the surrounding text color
unless the theme paints them; empty stars are transparent so the SVG
underlay (theme-provided or default outline) shows through.

| Token | Default | Paints |
|---|---|---|
| `--pkiw-star-rating-fill` | `currentcolor` | Filled portion of stars (e.g., the yellow part). Inherits surrounding text color by default. Themes set to `var(--wp--preset--color--accent)` or a literal value (`gold`, `#ffc107`) when a specific rating color is desired. |
| `--pkiw-star-rating-empty` | `transparent` | Empty portion of stars. Default transparent means only the SVG outline (or whatever the theme paints) shows; not painted by the plugin. |
| `--pkiw-star-rating-caption` | `inherit` | "4 of 5 stars" text below the rating. Default same color as surrounding body text. |
| `--pkiw-star-rating-border-radius` | `0` | Star corner radius. Was `2px` for a slight roundedness; default `0` keeps SVG paths' native shape. |
| `--pkiw-star-rating-variant-1` | `currentcolor` | Color modifier 1. Plugin currently uses `#e91e63` (pink) for one variant; tokenize for theme override. |
| `--pkiw-star-rating-variant-2` | `currentcolor` | Color modifier 2. Plugin currently uses `#2196f3` (blue) for one variant; tokenize for theme override. |

### Star rating exemption

The system font stack `-apple-system, BlinkMacSystemFont, "Segoe UI Symbol", system-ui, sans-serif` used in `src/blocks/star-rating/{style,editor}.css` is **NOT a token** and **stays as-is**.

Reason: this is a Unicode-coverage choice (many fonts don't have the
correct glyphs for star characters), not a design choice. Themes that
override the font would risk star glyphs rendering as boxes or fallback
characters.

If a theme wants different rating glyphs, it should override the
component's content via `::before` or a SVG override — not the
font-family.

**Phase D semantic clarification needed.** The three currently-used greens

## Interactivity state tokens

These control hover/active/disabled state intensities. Default values

| Token | Default | Paints |
|---|---|---|
| `--pkiw-state-hover-opacity` | `0.8` | Opacity of hovered cards. |
| `--pkiw-state-active-opacity` | `1` | Opacity of currently-active card. |
| `--pkiw-state-disabled-opacity` | `0.5` | Opacity of a disabled control. |

## Tokens NOT in this catalog (admin chrome — exempt)

Per Section 5.6 / Section 5.2's contract scope, the following surfaces
are NOT covered by `--pkiw-*` tokens. Hard-coded values in these surfaces
are intentional integration with WordPress admin chrome:

- `includes/admin/*.php` — admin pages, settings, import-page, webhooks-page, syndication-page, api-settings. Use WP admin colors (`#2271b1` admin blue, `#d63638` admin red, `#dba617` admin warning, `#46b450` admin success, `#646970` admin grey, `#c3c4c7` admin border, etc.) per `wp-admin/css/colors/*.css`.
- `admin/css/admin.css` — admin asset CSS. Same exemption.

This exemption is documented in `THEME-INTEGRATION.md` and enforced by
the C5 lint test (`tests/unit/test-no-color-leakage.php`) which excludes
`includes/admin/` and `admin/css/` paths from the leakage check.

## Token namespace contract

| Plugin | Prefix | Status |
|---|---|---|
| post-kinds-for-indieweb | `--pkiw-*` | this catalog |
| post-formats-for-block-themes | `--pfbt-*` | (sibling — see PFBT docs) |
| link-extension-for-xfn | `--xfn-*` | (sibling — XFN currently has no token namespace) |

Per Section 8.3 of the audit prompt, **no plugin reads or writes another's
tokens.** Themes set whichever they care about. If a theme wants the
listen card and a PFBT format card to share a background, the theme sets
both `--pkiw-listen-card-bg` and `--pfbt-format-link-bg` to the same
value — the plugins themselves never co-reference.

## Open follow-ups for C3 / C5

- **C3 migration order:** start with `src/blocks/shared/card-editor.css` (126 violations, propagates fix to 14 cards). Then per-block stylesheets, then patterns, then render-callbacks.
- **C3 `block.json` style attrs:** Section 5.5 says strip `backgroundColor`, `style.color.*`, `style.spacing.*` from pattern markup. The C1 inventory found 0 hits in `block.json` files but pattern PHP files do have inline style attributes — those need replacing with CSS-var-based markup OR moved to CSS rules.
- **C4 star-rating:** the contract requires keeping the SVG structure and font stack while only tokenizing the colors. The shape-keeping work goes alongside the token migration.
- **C5 `test-no-color-leakage.php`:** scans plugin CSS/PHP/JS for hex/rgb/named-color and fails the build if found outside `tests/fixtures/` AND outside the documented admin-chrome exemption (`includes/admin/`, `admin/css/`). Tokens themselves don't trigger leakage warnings because they ARE the documented values. The lint test enforces "non-admin code only references colors via `var(--pkiw-*)` lookups."
