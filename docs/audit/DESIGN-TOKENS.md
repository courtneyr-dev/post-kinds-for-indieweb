# Post Kinds for IndieWeb — Design Tokens Reference

Generated by audit Session C2, 2026-05-01.
**C3 migration + C5 enforcement completed 2026-07-03** — `styles/kind-tokens.css`
is enqueued (as a dependency of every block style handle), all non-admin
block CSS, block JS, and pattern markup reference colors only through
`var(--pkiw-*)` lookups, and `tests/phpunit/unit/NoColorLeakageTest.php`
enforces the contract in CI. (The C5 plan named this file
`tests/unit/test-no-color-leakage.php`; it was created following the
repo's actual PHPUnit convention instead.)

This is the authoritative per-token reference. Every token defined in
`styles/kind-tokens.css` is documented here with three columns: **token
name**, **what it paints** (which CSS property in which selector), and
**what happens if you don't set it** (the neutral default and what the
result looks like to the user).

The hard contract from Section 2 of the audit prompt:

> Plugin owns layout. Theme owns paint. The plugin's structural CSS may
> only reference colors and fonts through `var(--pkiw-*, neutral-fallback)`.
> If a theme sets the variable, the plugin uses it. If not, the kind
> inherits the theme's normal styling. **No kind ever forces a color the
> theme didn't ask for.**

If you remove all `--pkiw-*` variable definitions from your theme, the
plugin still renders correctly: the 14 h-cite card kinds inherit theme
typography with no painted background, while the surfaces that have
always shipped painted (checkin dashboard, venue detail, checkins feed,
media lookup, star rating) default to the theme palette with their
pre-2.0 colors as fallbacks — see "Bridge decision (2026-07-03, final)"
below. That's the intended baseline.

## How to use this catalog

### From a theme

Set tokens in your theme's `:root` or in section-scoped selectors:

```css
:root {
  --pkiw-listen-card-bg: var(--wp--preset--color--surface);
  --pkiw-listen-card-accent: var(--wp--preset--color--accent);
  --pkiw-star-rating-fill: gold;
}

/* Per-context override */
.is-style-condensed .pkiw-listen-card {
  --pkiw-listen-card-bg: transparent;
}
```

### From a plugin or pattern

Always reference tokens via `var()` with the documented neutral fallback:

```css
.pkiw-listen-card {
  background: var(--pkiw-listen-card-bg, transparent);
  color:      var(--pkiw-listen-card-fg, inherit);
}
```

If the fallback in the consumer's CSS doesn't match the default in
`kind-tokens.css`, the documented default wins (because tokens cascade
from the layer registration).

opinionated sizing that the plugin's grid logic depends on. Themes can
still override them when needed.

| Token | Default | Paints |
|---|---|---|

## Shared card paint tokens (default neutral)

These are the base tokens. Per-kind tokens default to these, so themes
can paint cards uniformly by setting only `--pkiw-card-*`, OR per-kind
by also setting `--pkiw-{kind}-card-*`.

| Token | Default | What you see if you don't set it |
|---|---|---|
| `--pkiw-card-bg` | `transparent` | No painted card background. Card text appears directly on theme body background. |
| `--pkiw-card-fg` | `inherit` | Card text uses surrounding text color (theme body text). |
| `--pkiw-card-accent` | `currentcolor` | Decorative accents (eyebrow, separator) match surrounding text. |
| `--pkiw-card-border` | `transparent` | No visible card border. |
| `--pkiw-card-border-radius` | `0` | Sharp corners. |
| `--pkiw-card-shadow` | `none` | No drop shadow. |
| `--pkiw-card-text-secondary` | `inherit` | Captions, dates, byline text — same color as body text by default. |
| `--pkiw-card-hover-bg` | `transparent` | Hover highlight on interactive rows/cards. |
| `--pkiw-card-hover-shadow` | `none` | Hover elevation shadow on feed items and dashboard cards. |

Sizing knobs `--pkiw-card-body-font-size`, `--pkiw-card-meta-font-size`,
and `--pkiw-card-icon-font-size` are consumed with per-context fallbacks
and intentionally have no catalog default — setting them overrides every
consumer at once.

## Star rating tokens (Section 5.4 special case)

The rating component currently hard-codes `#ffc107` (gold) for filled
stars. Token contract: filled stars inherit the surrounding text color
unless the theme paints them; empty stars are transparent so the SVG
underlay (theme-provided or default outline) shows through.

| Token | Default | Paints |
|---|---|---|
| `--pkiw-star-rating-fill` | `#ffc107` | Filled portion of stars. Semantic gold — intentionally palette-independent (bridge decision). Themes override for a different rating color. |
| `--pkiw-star-rating-empty` | `#ddd` | Empty portion of stars. Icons are font glyphs, so a concrete light grey keeps them legible on any surface (the C2 `transparent` spec would make them invisible). |
| `--pkiw-star-rating-caption` | `inherit` | "4 of 5 stars" text below the rating. Default same color as surrounding body text. |
| `--pkiw-star-rating-border-radius` | `0` | Star corner radius. Was `2px` for a slight roundedness; default `0` keeps SVG paths' native shape. |
| `--pkiw-star-rating-variant-1` | `#e91e63` | Hearts style fill. Semantic pink, palette-independent; theme-overridable. |
| `--pkiw-star-rating-variant-2` | `#2196f3` | Circles style fill. Semantic blue, palette-independent; theme-overridable. |

### Star rating exemption

The system font stack `-apple-system, BlinkMacSystemFont, "Segoe UI Symbol", system-ui, sans-serif` used in `src/blocks/star-rating/{style,editor}.css` is **NOT a token** and **stays as-is**.

Reason: this is a Unicode-coverage choice (many fonts don't have the
correct glyphs for star characters), not a design choice. Themes that
override the font would risk star glyphs rendering as boxes or fallback
characters.

If a theme wants different rating glyphs, it should override the
component's content via `::before` or a SVG override — not the
font-family.

## Stats block tokens — REMOVED (C2 audit error)

The C2 audit read `#127925`, `#127916`, and `#128214` in
`class-media-stats.php` / `class-now-playing.php` as "three close green
hex colors" and created `--pkiw-stat-positive/neutral/active` for them.
They are not colors: they are decimal HTML character references for the
kind icons — `&#127925;` 🎵, `&#127916;` 🎬, `&#128214;` 📖. Those files
contain no color literals at all, so the three tokens (and the "Phase D
three-greens consolidation" question) were removed at C3 (2026-07-03).
`NoColorLeakageTest` uses a negative lookbehind for `&` so character
references are never flagged as colors again.

## Tokens NOT in this catalog (admin chrome — exempt)

Per Section 5.6 / Section 5.2's contract scope, the following surfaces
are NOT covered by `--pkiw-*` tokens. Hard-coded values in these surfaces
are intentional integration with WordPress admin chrome:

- `includes/admin/*.php` — admin pages, settings, import-page, webhooks-page, syndication-page, api-settings. Use WP admin colors (`#2271b1` admin blue, `#d63638` admin red, `#dba617` admin warning, `#46b450` admin success, `#646970` admin grey, `#c3c4c7` admin border, etc.) per `wp-admin/css/colors/*.css`.
- `admin/css/admin.css` — admin asset CSS. Same exemption.

This exemption is enforced by the C5 lint test
(`tests/phpunit/unit/NoColorLeakageTest.php`), which excludes
`includes/admin/` and `admin/` paths from the leakage check. The same
reasoning extends to `src/editor/` (kind-selector sidebar chrome): those
components render in the WP admin sidebar, outside the themed canvas,
and mirror core component colors, so they are not scanned. Two raw
literals are individually allowlisted in the test with reasons (Leaflet
popup text; a `--wp-components-color-accent` fallback). A referenced
`THEME-INTEGRATION.md` was never written; this file is the authority.

Per Section 8.3 of the audit prompt, **no plugin reads or writes another's
tokens.** Themes set whichever they care about. If a theme wants the
listen card and a PFBT format card to share a background, the theme sets

value — the plugins themselves never co-reference.

## Bridge decision (2026-07-03, final)

The strict-neutral defaults shipped by the C3 migration were reversed the
same day by Courtney's decision: **the `--pkiw-*` token layer stays, but
catalog defaults for surfaces that shipped with concrete paint follow the
active theme palette with the pre-2.0 hex values as fallbacks**
(`var(--wp--preset--color--*, previous-hex)`). Nothing regresses visually
on themes that haven't adopted the tokens.

What this means in practice:

- The 14 h-cite card kinds shipped unpainted — their shared and per-kind
  tokens keep **neutral** defaults. No change for them.
- Checkin dashboard, venue detail, checkins feed, media lookup, and star
  rating shipped with concrete colors — their tokens now default to
  palette presets (`primary`/`contrast`/`tertiary`/`base`, plus the
  legacy `background` slug where the block historically used it) with
  the original hexes as fallbacks. Venue detail, checkins feed, and
  media lookup got their own per-block token sets
  (`--pkiw-venue-detail-*`, `--pkiw-checkins-feed-*`,
  `--pkiw-media-lookup-*`) so their concrete defaults don't leak onto
  the neutral shared `--pkiw-card-*` set.
- Star rating colors are semantic and palette-independent by design:
  fill `#ffc107`, empty `#ddd`, variant-1 `#e91e63`, variant-2
  `#2196f3`; only the caption follows the palette.
- `NoColorLeakageTest` no longer scans `styles/kind-tokens.css` — the
  catalog is the documented home of default paint values. All consumers
  remain literal-free and fully scanned.

Themes override exactly as before: set any `--pkiw-*` token at `:root`.

The C3 migration briefly shipped these with strict-neutral defaults;
the same-day bridge decision (see "Bridge decision" above) restored

with paint. The pass's decision framework below is kept for history.

Independently of that decision, the remaining hard-coded hex colors in the
block stylesheets (91 literals across 15 files) were audited on 2026-07-03
using the decision framework from the PFBT variation audit. The approach
is softer than the C2 neutral contract: colors default to the active
theme's palette presets and keep the previous hex values as fallbacks, so
nothing changes visually on themes without those preset slugs.

### Decision rules applied

1. **Front-end color that sits on themed output** → `var(--wp--preset--color--*, previous-hex)`.
   Slugs used: `contrast` (text and borders), `base` (card/dropdown
   surfaces), `tertiary` (muted surfaces like map placeholders and empty
   states), `primary` (links/accents), `background` (kept where the file
   already used it).
2. **Editor UI that mirrors front-end content or sits directly on the
   iframed canvas** (WP 7.0 themes the canvas) → same presets, so text
   stays readable on dark-palette themes.
3. **Self-contained editor chrome** — media placeholders (grey box +
   dashed `#ccc` border + `#666` label) and text/background chip pairs
   (`#757575` on `#f0f0f0`, admin blue on `#e7f3ff`) — stays hard-coded:
   the pair travels together and mirrors core's palette-independent
   `.components-placeholder`.
4. **Semantic rating colors** — star gold `#ffc107`, empty `#ddd`, heart
   pink, circle blue — stay palette-independent by design, but are all
   exposed as overridable custom properties.
5. **Leaflet popup text** stays hard-coded (popups are always white).
